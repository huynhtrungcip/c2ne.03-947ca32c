## Architecture of the demo data

Three data layers stack up on the dashboard:

| Layer | Source | Time window | Purpose |
|-------|--------|-------------|---------|
| **Historical baseline** | `src/data/historicalDataset.ts` (deterministic, ~600 events) | 2026-04-20 → 2026-04-24 | Tells the 5-day APT story so AI can refer to "this IP has been profiling us for 5 days". |
| **SQLite mirror** *(optional)* | `demo/seed-history/seed-historical-data.js` | same | Same data inserted into `server/soc_events.db` so backend APIs/AI tools can query it. |
| **Live NIDS stream** | Suricata + Zeek shippers → `server/index.js` | 2026-04-25 from ~09:00 | Real packets generated by `day6-2026-04-25-LIVE-attack.sh`, reshaped by backend into the 10 ML classes. |

> **Single source of truth:** the 5-day baseline is stored in
> `src/data/historicalDataset.ts`. The frontend always seeds from it on
> first load (idempotent). The SQLite seeder is a 1:1 mirror for the
> backend; both are safe to run multiple times.

the historical events represent. They exist for board audit ("you can
prove these events are realistic") but are **not** executed during the

## Quick start

### On the SOC server (this repo)

```bash
# 1. Pull the dashboard up
docker compose up -d
# 2. (Optional) mirror the 5-day baseline into the backend DB
node demo/seed-history/seed-historical-data.js
# 3. Open the dashboard → time range "All" or "Last 7d" → 5-day history visible
```

### On Kali (192.168.168.23)

```bash
# Rotate to the demo IP (only needed once before 25/04)
sudo ./demo/kali-attack/set-static-ip.sh 23

# On the morning of 25/04, run the live attack
sudo ./demo/kali-attack/day6-2026-04-25-LIVE-attack.sh
```

### Suricata

Drop `demo/suricata-rules/local.rules` into `/etc/suricata/rules/` on
the NIDS box and add it to `suricata.yaml → rule-files:`. Reload:

```bash
suricata-update && suricatasc -c reload-rules
```

## Two key implementation details

1. **Backend reshaper** (`server/index.js → reshapeForDemo`): every alert
   whose `src_ip` is the demo IP (`DEMO_ATTACKER_IP=192.168.168.23`) is
   rewritten so `attack_type` maps onto exactly one of the 10 ML
   classes and `confidence ≥ 0.92`. Real-world signatures from any
   other IP pass through unchanged.

2. **AI tool `get_ip_history`** (`src/lib/socTools.ts`): the analyst
   chatbot can query the full 5-day timeline of any IP. When the AI is
   asked "analyse 192.168.168.23" on 25/04, it will return both the
   live activity and the staging history → enabling the
   "this is APT staging" narrative.

See **ATTACK_STORYLINE.md** for the full narrative, and
**DEMO_RUNBOOK.md** for the minute-by-minute presentation script.
